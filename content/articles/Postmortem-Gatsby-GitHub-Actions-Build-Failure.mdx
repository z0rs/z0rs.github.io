Alright, quick story time.

This repo hit a full-on CI spiral. Local was chill, but GitHub Actions kept faceplanting on `gatsby build` with a new
error every round. One fix in, two new bugs out. Classic.

This post is the full timeline of what broke, why it broke, and every patch we shipped to get this thing stable again.

## What Went Sideways

### Symptom #1

```bash
Cannot query field "allMdx" on type "Query"
```

`createPages` was querying `allMdx`, but CI schema state wasn’t always lining up.

### Symptom #2

```bash
Error [ERR_REQUIRE_ESM]: require() of ES Module .../remark-gfm/index.js not supported
```

We had config/runtime mismatch. `remark-gfm` is ESM-only now, but part of the setup was still effectively going CJS
route.

### Symptom #3

```bash
The file 'gatsby-config' has both .js and .mjs variants... Using .js by default.
```

That one was nasty. Once Gatsby picked `.js`, schema got super thin in CI:

- `Cannot query field "childImageSharp" on type "File"`
- `Cannot query field "name" on type "SiteSiteMetadata"`
- `Cannot query field "defaultImage" on type "SiteSiteMetadata"`
- `Cannot query field "keywords" on type "SiteSiteMetadata"`

Also node count crashed (`Total nodes: 23`). Big red flag.

## Root Cause (No Fluff)

This was multi-layer, not one bug:

1. Page creation depended on a GraphQL query path that was brittle during CI bootstrap.
2. ESM vs CJS config behavior was mixed.
3. Dual Gatsby config variants (`.js` + `.mjs`) created non-deterministic config selection.
4. Workflow setup around Pages/configuration made that conflict easier to trigger.

## Full Fix Timeline

### 1) Stabilize Build Flow in `gatsby-node.js`

We moved page creation away from the brittle query path and hardened schema handling.

- `createPages` moved to `getNodesByType('Mdx')` + safety guards.
- `createSchemaCustomization` was tightened so MDX fields used by queries are explicit.

Why this mattered: CI bootstrap became way less fragile.

### 2) Normalize Workflow Behavior

In GitHub Actions:

- dependency install locked with `yarn install --frozen-lockfile`
- insecure TLS override removed

Why this mattered: reproducible install + cleaner runtime.

### 3) Force ESM Gatsby Config Path

We standardized on `gatsby-config.mjs` for ESM-native imports (`remark-gfm`, `rehype-*`).

Why this mattered: killed the `ERR_REQUIRE_ESM` path.

### 4) Stop Config Override/Collision in CI

Workflow was hardened so Gatsby doesn’t get tricked into loading a stale/extra config variant.

- removed static generator override behavior that could interfere with Gatsby config expectations
- added guard step to enforce `gatsby-config.mjs`
- auto-remove `gatsby-config.js` if it appears in runner workspace

Why this mattered: no more `.js` vs `.mjs` roulette.

### 5) Post-Incident UX Follow-up

After CI was green, we fixed article sidebar featured image behavior:

- landscape image stays landscape
- portrait image stays portrait
- no awkward crop vibe above “Share on LinkedIn”

That patch was not core CI infra, but it was part of the same stabilization sprint.

## What We Learned

1. Local success means nothing if CI boots differently.
2. Gatsby config must be single-source-of-truth. Pick one format and lock it.
3. Schema inference is cute until CI goes chaotic. Critical fields should be explicit.
4. CI workflow tools can mutate behavior in subtle ways, so treat workflow config like production code.

## Final Take

This incident was basically a chain reaction: config format mismatch, schema fragility, then workflow side effects
stacking on top. Once we treated it as a systems problem instead of a single error string, fixes became straightforward
and durable.

Now the pipeline is back to acting normal.
