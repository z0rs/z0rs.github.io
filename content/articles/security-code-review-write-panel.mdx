## **FINDINGS**

### **🔴 HIGH SEVERITY**

#### **1. Race Condition in GitHub API: Missing SHA Sync Between Check and Write**
- **File:** `src/api/write.js` lines 397–412 (publishToGitHub)
- **File:** `src/api/write-image.js` lines 251–265 (uploadToGitHub)
- **Issue:** Both functions perform an existence check (`GET /repos/.../contents/...`), then later commit the file. Between these two calls, the SHA can become stale if another process writes the same file.
  - Line 399–402: Fetches SHA
  - Line 405–411: Tries to commit with that SHA
  - **Gap**: If commit fails with `422 Unprocessable Entity` ("sha doesn't match"), it's caught as generic error (line 415–416) without retry or recovery.
- **Impact:** Silent commit failure, user sees success message, article isn't actually written to GitHub.
- **PoC:** 
  1. User A starts editing article X
  2. User B commits article X first
  3. User A commits → SHA mismatch → silently fails → no webhook trigger → site doesn't rebuild
  4. User A sees success message; article is not published.
- **Evidence:** Lines 414–417 catch all non-2xx statuses as terminal errors without distinguishing 422 (sha conflict) from 500 (server error).

#### **2. GitHub Token Validation Missing in Auth Check**
- **File:** `src/api/write.js` lines 92–125 (validateAuth) and `src/api/write-image.js` lines 87–120 (validateAuth)
- **File:** `.github/workflows/gatsby.yml` lines 9–12 (permissions)
- **Issue:** 
  - The `validateAuth` function checks if `WRITE_SECRET` matches the bearer token in production, but does **not verify that the token is used for GitHub API calls**.
  - The workflow permission block (line 9–12) specifies `contents: read` and `pages: write` but **workflows:write is missing**.
  - Line 336–338 in `write.js` dispatches the `gatsby.yml` workflow, but the token might only have `contents:write` permissions, causing workflow dispatch to silently fail.
- **Impact:** 
  - Article commits succeed, but site rebuild never triggers.
  - No error feedback to user; they see "rebuild triggered" in the response message (lines 428–433) even though it failed.
- **GitHub Requirement:** A PAT with `contents:write` + `workflows:write` is required per documentation (README line 205), but there's no **validation** that the provided token has both.
- **Evidence:**
  - Line 336: `/repos/z0rs/z0rs.github.io/actions/workflows/gatsby.yml/dispatches` requires `workflows:write`
  - Line 341: The return only checks `status === 204 || status === 200`, no handling for `403 Forbidden`
  - Lines 419–432: Response message claims rebuild triggered without checking actual success

#### **3. Slug Normalization Bypass via URL Encoding**
- **File:** `src/api/write.js` lines 144–167 (normalizeArticleSlug)
- **Issue:** The slug normalization regex `slug.includes('..')` (line 164) checks for literal `..` but doesn't handle **URL-encoded variations**:
  - `%2e%2e` (URL-encoded `..`) passes validation
  - `%252e%252e` (double-encoded `..`) passes validation
  - These are decoded later by GitHub API or filesystem operations, creating path traversal
- **PoC:**
  1. POST `/api/write` with `sourceSlug: "../../malicious%2e%2emdx"`
  2. Normalization passes (no `..` literal found)
  3. GitHub API or filesystem decodes it → writes outside `content/articles/`
- **Evidence:** 
  - Line 164 is a simple string contains check, not a decoded URL check
  - Line 158–161 parse tags from YAML but don't decode frontmatter values
  - `normalizeArticleSlug` is called on raw input without decoding first (line 254, 361, 485)
- **Impact:** File write outside intended directory; potential RCE if written to `.github/workflows/` and deployed.

#### **4. XSS via Frontmatter Tags in Markdown Snippet**
- **File:** `src/api/write-image.js` lines 198–201 (buildMarkdownSnippet)
- **Issue:** The alt text passed to `buildMarkdownSnippet` comes from user input (`body?.alt`) and is not escaped before being inserted into markdown.
  - Line 199: `const safeAlt = typeof alt === 'string' && alt.trim() ? alt.trim() : 'image';`
  - "safeAlt" is only type-checked and trimmed, **not HTML-escaped or markdown-escaped**.
  - Line 200: Directly interpolates into markdown: `![${safeAlt}](${publicUrl})`
- **PoC:**
  1. Upload image with alt text: `](http://evil.com/xss.js) <img src=x onerror="alert(1)">`
  2. Markdown snippet: `![](http://evil.com/xss.js) <img src=x onerror="alert(1)">/images/uploads/...`
  3. When rendered in browser, this becomes executable HTML
- **Impact:** XSS in article content; attacker can steal session tokens, deface articles.
- **Evidence:**
  - No markdown escaping in `buildMarkdownSnippet`
  - Client copies this snippet directly into textarea (user sees the raw markdown)
  - When article is published, the markdown is base64-encoded to GitHub (safe there), but if ever rendered via unescaped HTML, XSS triggers

#### **5. No Validation of Draft vs Published Status at Read Time**
- **File:** `gatsby-node.js` lines 207–222
- **Issue:** The build-time page creation filters articles by `status !== 'draft'` (line 207), but:
  - There's no enforcement that draft articles cannot be accessed via direct URL
  - If a draft article's slug is known (e.g., from a sitemap or Git history), it's still rendered
  - The write API doesn't validate that drafts cannot be published (line 240 in `write.js` only sets `status` field, doesn't enforce it)
- **PoC:**
  1. Create draft article: `my-secret-draft`
  2. Attacker somehow knows the slug
  3. Direct visit to `/articles/my-secret-draft/` → 404 (not in build)
  4. But if someone commits a published version later, it's exposed
- **Impact:** Information disclosure; draft articles could be leaked if the write panel is compromised.
- **Evidence:**
  - Line 207: Only filters at build time, not at runtime
  - No `robots: noindex` or 403 response for draft articles
  - Drafts are published to GitHub (commits exist in history)

### **🟡 MEDIUM SEVERITY**

#### **6. Bearer Token Accepted as "Any Non-Empty String" in Dev Mode**
- **File:** `src/api/write.js` lines 96–97 and `src/api/write-image.js` lines 91–92
- **Issue:** In local dev (`NODE_ENV !== 'production'`), any non-empty bearer token is accepted.
  - Line 97 in `write.js`: `if (bearer) return { valid: true };`
  - This means `Authorization: Bearer x` or `Authorization: Bearer nonsense` both work
  - No distinction between intended token and accidental/spoofed token
- **Impact:** Local dev is **not a security boundary**, but if dev server is exposed to network, anyone can use the write API.
- **Residual Risk:** Dev deployments to cloud runners (e.g., GitHub Codespaces) might expose the write API.
- **Evidence:** Line 96–104 in both files; no token comparison in dev mode.

#### **7. GITHUB_TOKEN Not Validated Before Use in GitHub API Calls**
- **File:** `src/api/write.js` lines 598–604, 642–648, 699–705
- **File:** `src/api/write-image.js` lines 306–312
- **Issue:** Token is fetched from environment but never tested:
  - No HEAD request to validate token format or permissions
  - If token is missing, expired, or revoked, error is thrown at publish time (line 415 in `write.js`)
  - User has already written content to form, triggers publish, then gets an error
- **Impact:** Poor UX; no pre-validation of GitHub token; potential for state inconsistency.
- **Fix:** Add token validation on API initialization or during first GitHub call.

#### **8. No Rate Limiting on Write/Delete Operations**
- **File:** `src/api/write.js` (entire handler) and `src/api/write-image.js` (entire handler)
- **Issue:** Any client with a valid token can:
  - POST unlimited articles without rate limiting
  - DELETE unlimited articles without soft-delete or restore
  - Each action triggers a workflow dispatch (potential GitHub Actions quota abuse)
- **Impact:** DoS via token compromise; attacker can spam articles or mass-delete all content.
- **Evidence:** No rate limiting middleware or counter in the handlers.

#### **9. Silent Failure in Image Upload if GitHub Commit Fails**
- **File:** `src/api/write-image.js` lines 303–327
- **Issue:**

  - Line 314–319: Calls `uploadToGitHub`, which throws on failure (line 263)
  - Line 321–327: On success, returns markdown snippet with image URL
  - **But**: If upload fails (exception in line 314), catch block (line 342–344) returns generic error without logging which step failed

  - URL returned in response might be wrong if GitHub call partially succeeded
- **Impact:** Broken image URLs in articles; silent failures.
- **Evidence:** Line 342–344 catches all errors uniformly.

### **🟢 LOW SEVERITY**

#### **11. Missing Content-Length Validation on Image Uploads**
- **File:** `src/api/write-image.js` lines 159–176
- **Issue:**
  - Image payload is decoded from base64 (line 161), then validated for size (line 169)
  - **But**: There's no validation of `body` size before base64 decoding
  - If client sends huge base64 string, decoding happens first, consuming memory
  - No header-level Content-Length check (though `Content-Length: Buffer.byteLength(rawBody)` is set in `githubFetch`, line 218)
- **Impact:** Memory exhaustion attack; Denial of Service via large base64 payloads.
- **Evidence:**
  - Line 161: `Buffer.from(base64, 'base64')` can allocate large memory
  - No limit on `dataUrl` string length before decoding
  - Line 169: Size check happens after decoding

#### **15. YAML Frontmatter Parsing is Naive**
- **File:** `src/api/write.js` lines 219–251
- **Issue:**
  - Regex-based parsing at lines 222: `/^---\n([\s\S]*?)\n---\n?/`
  - Doesn't handle edge cases:
    - Extra newlines: `---\n\n---\n` (captures empty frontmatter)
    - YAML comments: `# this is ignored` (parsed as key "# this")
    - Multiline values: `description: |` (broken; only line 1 captured)
    - Duplicate keys: Only last value kept (line 237)
  - Line 229–238: Split by newline and parse key-value manually; no YAML parsing library
- **Impact:** Malformed YAML can be silently accepted and serialized back incorrectly.
- **PoC:** Upload article with:
  ```yaml
  title: My Article
  tags: [tag1, tag2
  author: Eno
  ```
  Frontmatter is truncated; tags field is corrupted.
- **Evidence:** Lines 219–251 do manual string parsing instead of using `yaml` library.

#### **17. No Filesystem Symlink Protection**
- **File:** `src/api/write.js` lines 507–515 (writeToFilesystem) and lines 484–505 (readFromFilesystem)
- **Issue:**
  - `join(process.cwd(), CONTENT_PATH, filename)` doesn't validate that the result is still within `CONTENT_PATH`
  - If `filename` is a symlink (e.g., `../../secret.mdx`), it could write outside the intended directory
  - While `normalizeArticleSlug` attempts to block `..`, it doesn't prevent symlink-based escape
- **Impact:** Filesystem escape attack; attacker could write to arbitrary locations.
- **PoC (requires pre-existing symlink):**
  1. Attacker creates symlink: `ln -s /tmp/malicious.mdx content/articles/escape`
  2. Write article with slug `escape`
  3. File is written to `/tmp/malicious.mdx` instead of `content/articles/escape.mdx`
- **Evidence:** No `realpath` validation in lines 509, 536.

## **FOLLOW-UP REFACTORS** (Non-Urgent)

1. **Extract shared utilities:** `getRuntimeEnv`, `getHeader`, `parseBodyValue`, `validateAuth` are duplicated in `write.js` and `write-image.js`. Move to `src/lib/auth.js`.

2. **Replace regex YAML parsing with `yaml` npm package:** Current parsing is fragile (lines 219–251 in `write.js`). Use `import YAML from 'yaml'` for robust parsing.

3. **Structured logging:** Replace `console.warn` in `gatsby-node.js` with a logger that supports JSON output and structured fields.

4. **Implement idempotent writes:** Add SHA checking and retry logic to avoid conflicts. Consider using GitHub's `merge` endpoint for safer updates.

5. **Add request signing:** Use HMAC-SHA256 for webhook signature verification (if webhooks are added in future).

6. **Separate auth layer:** Extract `validateAuth` into a Gatsby function wrapper to reduce duplication and ensure consistent validation across endpoints.

## **CONCLUSION**

**No critical findings that immediately compromise the system, but several HIGH and MEDIUM issues that create silent failures, race conditions, and information disclosure.**

The codebase shows thoughtful security considerations (SSRF blocking in `gatsby-node.js`, slug normalization, auth gating), but has:
- **Race conditions** in GitHub API interactions (SHA sync)
- **Missing validation** of token permissions before use
- **Input encoding issues** (path traversal, XSS)
- **Silent failures** (no indication when rebuilds fail)
- **Poor observability** (no logging, error details leaked)

Recommend addressing **Rank 1 priorities** before production use, especially if multiple admins use the write panel simultaneously.
